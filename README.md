# FARD WIN DEMO

FARD WIN DEMO is a single-repo adoption demo that proves, with real receipts and replayable outputs, that FARD can solve painful day-to-day problems for three incumbent cohorts:

- Python users: analytics, reproducibility, visualization, deploy sprawl
- Rust users: startup cost, concurrency ceremony, systems-style throughput
- JavaScript users: full-stack drift, package chaos, supply-chain instability

This repository is both:

- a **working execution system**
- a **precise specification of package boundaries, outputs, and build order**

That dual role is intentional. The repo does not just describe the future system. It runs.

## Core claim

Every meaningful result in this repo is produced by actual FARD execution and anchored by SHA-256 receipts.

Same inputs -> same outputs -> same digests.

No mocked benchmarks. No simulated receipts. No fake artifacts.

## The four runnable demos

- `apps/py_pain_demo` — analytics, grouped summaries, deterministic plot artifact
- `apps/rust_pain_demo` — startup probe, JSON throughput probe, parallel pipeline
- `apps/js_pain_demo` — structured JSON flow, DOM construction, digest-backed output
- `apps/control_tower` — orchestration, summary generation, receipt collection, human-readable reporting

## Verification Challenge

Hash the generated Python demo plot:

```bash
shasum -a 256 out/py_demo_hist.png
```

Expected hash:

```text
9bb85ba1337c0363b8242924a171c47dfb73d34bac63cb40e7d3443f51b6da9a  out/py_demo_hist.png
```

If your hash matches, you have reproduced the same plot bytes and the same deterministic pipeline.
